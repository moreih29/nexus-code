# File Editing UX 패턴 — Cursor 레퍼런스 조사

> 작성일: 2026-03-27
> 목적: Nexus Code의 파일 편집 UX 설계를 위한 레퍼런스

---

## 1. 인터랙션 모델

### 1-1. 인라인 편집 (Cmd/Ctrl+K)

**What**
코드 선택 후 단축키를 누르면 에디터 내에 프롬프트 입력창이 나타난다. 자연어 명령을 입력하면 해당 위치에 인라인 diff가 표시되고, Accept/Reject로 확정한다.

**Why**
컨텍스트 스위칭 없이 편집 의도와 결과를 동일 뷰포트에서 처리. 작은 범위의 수정에서 인지 부하를 최소화한다.

**Trade-off**
- 좋음: 빠른 피드백 루프, 파일을 이탈하지 않음
- 나쁨: 복잡한 멀티파일 작업엔 부적합, 팝업 UI가 에디터 공간을 잠식

---

### 1-2. Chat Apply (사이드바 → Apply 버튼)

**What**
사이드바 채팅창(Cmd+L)에서 AI 응답 코드 블록 옆의 **Apply** 버튼을 클릭하면, 현재 열린 파일에 diff가 생성된다. diff를 확인한 후 수락하거나 거부한다.

**Why**
긴 대화 맥락이 필요한 작업에서 채팅 흐름을 유지하면서도 파일 수정을 명시적으로 트리거. AI 제안과 실제 파일 편집을 분리하여 사용자가 제어권을 유지한다.

**Trade-off**
- 좋음: 채팅 컨텍스트 유지, 명시적 Apply 단계로 실수 방지
- 나쁨: 단일 파일에 종속, 여러 파일 동시 편집엔 추가 단계 필요

---

### 1-3. Agent 모드 (멀티파일 자율 편집)

**What**
Agent가 여러 파일을 자율적으로 수정하고, 완료 후 통합 diff 뷰를 제공한다. 각 파일 탭 상단(breadcrumb 영역)에 **Keep File / Undo File** 버튼이 노출된다. "Review changes" 버튼으로 전체 변경 목록을 한번에 검토할 수 있다.

**Why**
복잡한 리팩토링이나 대규모 변경은 단일 파일 편집 루프로는 불가능하다. 자율 실행 후 일괄 검토 모델은 개발자가 진행 중 개입할 필요를 줄인다.

**Trade-off**
- 좋음: 대규모 변경을 일괄 검토, 파일 간 일관성 보장
- 나쁨: 자동 적용(최근 업데이트 이후)으로 검토 스텝이 약화될 수 있음, 변경 범위가 클수록 검토 피로도 증가

---

### 1-4. Tab 자동완성 (예측적 인라인 제안)

**What**
타이핑 중 AI가 멀티라인 제안을 회색 고스트 텍스트로 표시한다. Tab으로 수락, Esc로 거부, Ctrl+→ (⌘+→)로 단어 단위 부분 수락이 가능하다. 수락 후 다음 편집 위치로 자동 점프하여 "Tab-Tab-Tab" 연속 플로우를 지원한다.

**Why**
명시적 명령 없이 흐름을 유지하는 마이크로 편집 루프. 반복 패턴 변경 시 동일 변경을 파일 전체에 예측·제안하여 반복 작업을 자동화한다.

**Trade-off**
- 좋음: 흐름 중단 없음, 빠른 반복 작업에 최적
- 나쁨: 항상 켜져 있어 원치 않는 제안이 잦을 수 있음, 부분 수락 UX가 비직관적

---

## 2. 정보 밀도 / 계층

### Diff 표시 방식

- **인라인 diff**: 추가된 줄은 녹색, 삭제된 줄은 빨간색으로 에디터 내 원위치에 표시
- **컨텍스트 유지**: 변경된 줄 주변 코드(컨텍스트 라인)를 함께 보여주어 변경의 의미를 파악 가능
- **두 단계 AI 처리**: 주 LLM이 변경 스케치를 생성 → 별도 "Apply 모델"이 기존 코드에 지능적으로 통합. 단순 텍스트 매칭이 아니라 코드 구조를 이해하고 적용

### 멀티파일 집계 뷰

- Agent 완료 후 "Review changes" 버튼으로 전체 변경된 파일 목록을 집계 표시
- 파일 단위 순차 검토 ("Review next file" 버튼) 또는 파일 탭 클릭으로 해당 파일 diff로 이동

---

## 3. 상태 표시

| 상태 | UI 표현 |
|------|---------|
| AI 생성 중 | diff 라인이 실시간으로 나타남, Stop 버튼 노출 |
| 검토 대기 | 파일 탭 상단에 "Keep File / Undo File" 버튼, 녹색/빨간색 diff 강조 |
| 완료 (수락) | diff 강조 해제, 변경사항이 파일에 반영 |
| 완료 (거부/롤백) | 원본 상태로 복원, diff 강조 해제 |
| Agent 실행 중 | "Review next" / "Undo" / "Keep" 버튼이 동적으로 교체됨 (UX 이슈 보고됨) |

**롤백**: 파일 단위 Undo File로 AI 변경 전 상태로 복원 가능. 전체 세션 롤백은 불가 (파일별 처리).

## 4. 레이아웃

### 인라인 vs 사이드바이사이드

Cursor는 두 레이아웃을 상황에 따라 선택적으로 사용한다:

| 모드 | 레이아웃 | 사용 상황 |
|------|----------|-----------|
| Cmd+K | 인라인 (에디터 내 팝업) | 단일 파일 국소 편집 |
| Chat (Cmd+L) | 사이드바 + 에디터 diff | 맥락이 필요한 단일 파일 편집 |
| Agent | 사이드바 대화 + 다중 파일 탭 diff | 멀티파일 자율 편집 |

### 파일 네비게이션

- 변경된 파일 목록: 에디터 파일 탭에 시각적 표시
- "Review next file" 버튼으로 변경된 파일 순서대로 탐색
- 파일 탭 상단(breadcrumb 근처)에 Keep/Undo 컨트롤 위치 (최근 UI 변경)

## 5. 경쟁 도구와의 차별점 비교

| 특징 | Cursor | Windsurf (Cascade) | GitHub Copilot Edits |
|------|--------|--------------------|----------------------|
| 인라인 편집 | Cmd+K, 인라인 diff | Supercomplete (diff 박스) | 인라인 제안 Tab |
| 멀티파일 관리 | Agent 모드, 파일별 Keep/Undo | Cascade 메모리, 세션 누적 추적 | Working Set 정의 후 일괄 적용 |
| 컨텍스트 방식 | @파일명 수동 참조 | 세션 전반 자동 추적 (편집+터미널+클립보드) | Working Set 명시 정의 |
| diff 검토 | 인라인 + 집계 뷰 | 인라인 diff 박스 | 파일별 Accept/Reject |
| 자동화 수준 | YOLO 모드(검토 스킵) ~ 단계별 검토 | 세션 기반 자율 진행 | 명시적 제어 중심 |

**Cursor의 핵심 차별점:**
- Apply 모델 분리(LLM 스케치 → Apply 모델 통합)로 코드 통합 품질 향상
- Tab-Tab-Tab 연속 수락 플로우로 반복 작업 가속
- Plan Mode(Shift+Tab)로 실행 전 계획 검토 가능

## 6. Nexus Code 설계 시 적용 포인트

1. **명시적 Apply 단계 필수**: AI 제안이 자동 적용되지 않고, 사용자가 확인 후 적용하는 명시적 단계를 유지한다. (Cursor 최근 자동 적용 전환이 사용자 불만 야기)

2. **인라인 diff 우선**: before/after를 같은 뷰에서 비교할 수 있는 인라인 diff가 별도 diff 창보다 컨텍스트 유지에 유리하다.

3. **파일 단위 Keep/Undo**: 멀티파일 변경에서 파일별 수락/거부가 필요하다. 전체 일괄 수락은 보조 옵션으로 제공.

4. **상태 명확성**: AI 생성 중, 검토 대기, 완료 상태를 시각적으로 구분한다. 버튼이 동적으로 교체되면 의도치 않은 클릭을 유발한다(Cursor 포럼 이슈).

5. **Tab 자동완성의 부분 수락**: 단어 단위 수락(Ctrl+→)은 긴 제안을 다루는 데 실용적이다.

## 실현성 검토 (Architect)

> 현재 아키텍처: EditRenderer/WriteRenderer (인라인 diff 카드), HookServer manual 모드 pre-tool-use 가로채기, stream-json tool_call은 실행 후 도착

### 1-1. 인라인 편집 (Cmd/Ctrl+K)

- **실현 가능성**: 낮음
- **기술적 제약**: **근본적 제약** — Nexus Code는 에디터가 아닌 채팅 래퍼. 코드 선택 + 인라인 프롬프트 팝업은 에디터 코어(Monaco, CodeMirror 등)가 필요. 현재 아키텍처에서 파일 내용을 에디터로 열어 인라인 편집하는 기능은 없음.
- **필요 사항**: 에디터 통합 (Monaco 등) → 아키텍처 전환 수준의 변경.
- **구현 난이도**: Very High (아키텍처 변경)

### 1-2. Chat Apply (사이드바 → Apply 버튼)

- **실현 가능성**: 보통
- **기술적 제약**: 현재 Edit/Write 도구는 CLI가 자율적으로 실행. "Apply 버튼"으로 사용자가 명시적으로 트리거하려면 두 가지 방식이 가능: (1) HookServer manual 모드에서 Edit/Write pre-tool-use를 가로채고 PermissionCard에서 Apply/Reject UI 제공. (2) CLI 응답에서 코드 블록을 파싱하여 별도 Apply 메커니즘 구현 — 이 경우 CLI 외부에서 파일 쓰기를 Electron이 직접 수행해야 함.
- **필요 사항**: 방식 (1) 권장 — `PermissionCard`를 Edit/Write 전용으로 확장하여 diff 뷰 + Apply/Reject 버튼. `PermissionHandler.isAutoApproved()`에서 Edit/Write를 제외하여 항상 수동 승인.
- **구현 난이도**: Medium

### 1-3. Agent 모드 (멀티파일 자율 편집)

- **실현 가능성**: 보통
- **기술적 제약**: CLI Agent 도구가 멀티파일을 수정하면 각 Edit/Write tool_call이 순차적으로 도착. 현재 `session-store`가 toolCalls를 message 단위로 누적하므로, 한 턴 내 변경된 파일 목록 집계는 가능. 다만 **파일별 Keep/Undo**는 Electron에서 파일 백업/복원 인프라가 필요.
- **필요 사항**: (1) 턴 내 Edit/Write toolCalls를 파일별로 그룹화하는 셀렉터. (2) "Review Changes" 버튼 → RightPanel에 변경 파일 목록 + 파일별 diff 뷰. (3) Keep/Undo — 세션 시작 시 작업 디렉토리 git stash 또는 파일별 백업 (`.nexus-code/backup/`), Undo 시 복원.
- **구현 난이도**: 변경 집계 뷰 Medium, Keep/Undo High

### 1-4. Tab 자동완성

- **실현 가능성**: 낮음
- **기술적 제약**: 에디터 내 기능. Nexus Code에는 에디터가 없으므로 적용 불가.
- **구현 난이도**: N/A (아키텍처 범위 밖)

### 2. Diff 표시 방식

- **실현 가능성**: 높음
- **기술적 제약**: 없음. 현재 `EditRenderer`가 `old_string`/`new_string`을 빨간/녹색으로 인라인 diff 표시 중. 컨텍스트 라인 표시와 라인 번호는 추가 가능.
- **필요 사항**: (1) 라인 번호 표시 (offset 정보 활용). (2) unified diff 형식으로 개선 — `diff` 라이브러리 사용하여 hunk 단위 표시. (3) 멀티파일 집계 뷰는 위 1-3에서 다룸.
- **구현 난이도**: Low~Medium

### 3. 상태 표시

- **실현 가능성**: 높음
- **기술적 제약**: 없음. 현재 `resolveStatus()`가 running/done/error 판별. AI 생성 중(running) → 검토 대기 → 완료 상태 전이는 이미 동작 중.
- **필요 사항**: 검토 대기 상태를 시각적으로 강화 — Edit/Write가 manual 모드에서 PermissionCard로 표시될 때 diff + Accept/Reject UI.
- **구현 난이도**: Low (PermissionCard 확장 시 자연스럽게 해결)

### 4. 레이아웃

- **실현 가능성**: 보통
- **기술적 제약**: 현재 인라인 카드(채팅 내) + RightPanel(사이드 탭) 이중 구조. Cursor 스타일의 "사이드바 대화 + 에디터 diff"는 에디터 통합 없이는 불가. 다만 RightPanel에 diff 전용 탭을 추가하여 "채팅 인라인 요약 + 사이드 패널 full diff" 패턴은 가능.
- **필요 사항**: RightPanel에 "Changes" 탭 추가 → 현재 턴의 Edit/Write 변경 사항을 파일별 diff로 표시.
- **구현 난이도**: Medium
