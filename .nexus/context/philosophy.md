# Philosophy

## 프로젝트 정의

nexus-code는 여러 프로젝트를 오가며 작업하는 개발자를 위한 **단일 창 멀티 워크스페이스 IDE**다.
창 하나에서 워크스페이스를 즉시 전환하면서, 각 워크스페이스마다 독립된 AI 코딩 하네스가 네이티브 터미널 수준으로 동작하는 개발 환경을 제공한다.

---

## 핵심 4축

### 1. 단일 창 멀티 워크스페이스 전환

워크스페이스는 에디터 상태, 파일트리, 터미널, AI 세션, git 상태를 하나로 묶은 원자적 단위다.
사용자는 창을 새로 열거나 컨텍스트를 다시 구성하지 않고도 프로젝트 간을 즉시 전환한다.

### 2. VSCode급 능력 표면

편집, LSP, 레이아웃, 키바인딩은 사용자가 VSCode에서 기대하는 표준과 동등한 수준으로 맞춘다.
목표는 VSCode 전체 복제가 아니라, 1인 개발자가 끝까지 만들고 유지할 수 있는 능력 표면을 확보하는 것이다.

### 3. AI 하네스가 네이티브 터미널처럼 동작

claude-code, opencode, codex가 전용 터미널 에뮬레이터처럼 동작한다.
한글 입력 밀림, 커서 오위치, 조합 문자 소실 없이 작동하는 것을 보장한다.

### 4. 프로세스·포트 격리는 덤

워크스페이스 간 프로세스와 포트는 결과적으로 섞이지 않는다.
이 특성을 전면에 내세우지는 않지만, 설계상 자연스럽게 보장된다.

---

## Non-Goals

다음은 의도적으로 포기한 영역이다. 기능으로 일부 흡수될 수 있어도, **제품 정체성으로 삼지 않는다.**

- **AI 에이전트 비교·오케스트레이션 UX**: 여러 AI 에이전트를 한 화면에서 비교하거나 오케스트레이션하는 전용 UI는 만들지 않는다. 하네스는 플러그인 경계로 격리되며, IDE는 읽기 전용으로 관찰한다.
- **CJK 1등 시민 마케팅·슬로건**: 한국어·일본어·중국어 품질을 챙기되, 이를 포지셔닝 문구로 쓰지 않는다.
- **초기 단계 협업·원격·팀 기능**: MVP에서는 단일 사용자 로컬 환경에 집중한다. 원격 워크스페이스와 팀 공유는 포스트-MVP 영역이다.

---

## 기본 트레이드오프 선호

결정이 어려울 때 이 선호를 따른다.

- **검증된 경로 > 성능 극단**: 생태계에서 검증된 구현을 먼저 선택한다. 성능 병목이 실측으로 확인된 이후에 최적화한다.
- **실현성 > 야심**: 1인 개발자가 정해진 기간 안에 완성할 수 있는 범위를 지킨다. 야심찬 아이디어는 스코프 밖으로 명시적으로 내보낸다.
- **외부 API 변동에 정체성이 흔들리지 않는다**: 하네스 프로토콜이나 외부 API가 바뀌어도 제품 정체성은 유지된다. 외부 의존은 플러그인 레이어로 격리한다.

---

## 언어 정책

- **UI 언어**: 영어 단일. 한국어 UI 번역은 제공하지 않는다.
- **입력·렌더링·경로 처리 공식 보장**: 영어와 한국어. 한국어 품질 리그레션은 블록커 이슈로 분류한다.
- **일본어·중국어**: best-effort. 리그레션은 블록커가 아니다.

상세 체크리스트는 `korean-quality.md`에 기록한다.
