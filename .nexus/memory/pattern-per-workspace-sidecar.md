# pattern-per-workspace-sidecar

메인 UI 프로세스 1개와 워크스페이스당 독립 sidecar 프로세스 1개를 쌍으로 운영해, UI 재시작과 워크스페이스 생명주기를 완전히 분리한다.

## 적용 조건 — 언제 이 패턴이 맞는가

다음 조건을 **모두** 충족할 때 이 패턴을 선택한다.

- 여러 워크스페이스 간 전환 시 **끊김 없는 즉시 전환 UX**가 필수 요구사항이다(cmux 수준).
- 워크스페이스마다 장기 실행해야 하는 감독 대상이 존재한다(LSP 서버, AI 하네스, 파일 와처, PTY 프로세스 그룹 등).
- 워크스페이스 간 **완전 격리**가 필요하다(한 워크스페이스의 크래시·포트 충돌이 다른 워크스페이스에 파급되어서는 안 된다).
- UI 프로세스가 재시작되어도 워크스페이스의 실행 상태(터미널 스크롤백, 진행 중 스트림, AI 세션)가 유지되어야 한다.

다음 중 하나라도 해당하면 이 패턴을 **쓰지 않는다**.

- 동시에 열리는 워크스페이스 수가 항상 1개 이하다(sidecar 구조의 복잡도가 낭비다).
- 워크스페이스당 baseline 메모리 비용이 허용 불가한 제약 환경(예: 임베디드, 저사양 서버)이다.
- 워크스페이스 간 상태 공유가 필요한 기능이 핵심이다(예: 크로스 워크스페이스 심볼 인덱스).

## 장점

- **UI와 sidecar 생명주기 분리**: UI(Electron main 프로세스)가 재시작되거나 크래시해도 sidecar는 계속 실행된다. UI 재연결 시 sidecar의 기존 상태를 그대로 이어받는다.
- **즉시 전환 UX**: 비활성 워크스페이스의 sidecar가 백그라운드에서 살아있으므로(P1 정책), 전환 시 새 프로세스를 기동하는 지연이 없다. cmux의 패인 전환과 동일한 체감을 Electron UI에서 구현할 수 있다.
- **워크스페이스 격리**: 한 sidecar의 크래시, 메모리 누수, 포트 충돌이 다른 sidecar에 파급되지 않는다. OS 프로세스 경계가 격리 단위가 된다.
- **대화 이력 항상 보존**: AI 세션 파일은 sidecar 외부(디스크)에 기록되므로, sidecar가 종료되어도 `--resume <sessionId>`로 복원 가능하다(`pattern-per-turn-harness-spawn.md` 참조).

## 단점 · 트레이드오프

- **Baseline 메모리 선형 증가**: 열린 워크스페이스 수에 비례해 sidecar 프로세스 수가 늘어난다. Go 바이너리 기준은 Node.js 대비 meaningful하게 낮은 baseline을 기대할 수 있으나(연구 기반 추정, `external-bun.md` · `external-nodejs.md` 정성 비교 참조), 파워 유저 시나리오(동시 약 20개 이상의 워크스페이스를 상시 유지)에서는 누적 부담이 커져 P2 suspend 정책을 통한 idle 회수가 권장된다. 정확한 수치는 구현 후 실측으로 확정한다.
- **IPC 계약 관리 부담**: UI(TypeScript)와 sidecar(Go) 사이의 메시지 계약을 코드젠으로 동기화해야 한다. 계약 드리프트가 발생하면 무음 파손으로 이어진다.
- **sidecar 버전 관리**: 앱 업데이트 시 기존에 실행 중인 sidecar를 안전하게 교체하는 절차가 필요하다.

## Idle 정책 3단

| 정책 | 동작 | 적용 시점 |
|---|---|---|
| **P1 Always-on** (기본값) | 워크스페이스가 열려있는 한 sidecar를 계속 running | 모든 워크스페이스에 기본 적용 |
| **P2 Suspend** (옵션) | N시간 idle 후 `SIGSTOP` → 복귀 시 `SIGCONT`. CPU 0, 메모리 유지 | 설정으로 활성화. 다수(설계 가정상 약 20+)의 워크스페이스를 동시에 열어두는 파워 유저용 |
| **P3 Kill + --resume** | sidecar 종료, 메모리 해제. 재진입 시 새 sidecar + AI `--resume` | 사용자가 워크스페이스를 **명시적으로 닫을 때만** |

P3는 명시적 close 시에만 적용한다. idle timeout으로 자동 kill하지 않는다(터미널 스크롤백·진행 중 스트림 손실 위험).

## 대안과 비교 — 왜 이 패턴을 선택하는가

**M1 Hot-All**: 모든 워크스페이스의 모든 서비스를 항상 메모리에 올려둔다. 전환 속도는 최고지만 워크스페이스 수에 따라 메모리가 폭주한다. 현실적 워크스페이스 수(5-20개)에서 채택 불가.

**M2 Cold Switch**: 전환 시마다 이전 sidecar를 종료하고 새 sidecar를 기동한다. 메모리 사용은 최소지만 LSP 초기화·하네스 연결에 수 초가 소요돼 "즉시 전환" UX를 구현할 수 없다.

**M3 Hybrid LRU**: 최근 N개 워크스페이스만 hot, 나머지는 cold로 관리한다. 이론적으로 균형점을 찾을 수 있으나 LRU 캐시 크기 튜닝 부담이 1인 팀에서 유지 비용 대비 이득이 불분명하다.

**M4 Per-Workspace Sidecar (이 패턴)**: P1 기본 + P2 옵션으로 메모리와 UX의 균형을 사용자가 직접 선택하게 한다. cmux의 워크스페이스 전환 철학을 IDE 컨텍스트에서 가장 충실하게 구현하는 모델이다.
