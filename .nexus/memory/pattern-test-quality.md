# pattern: 테스트 품질 판별 기준

> 목적: 테스트 케이스를 "좋은 테스트 / 나쁜 테스트"로 판별하는 공통 잣대를 고정한다. 전수 점검과 신규 테스트 작성 모두에서 같은 기준을 쓴다.
> 적용 범위: `tests/**` 전체 (unit · integration).
> 연계: 목/DI 작성 규칙은 `pattern-bun-mock-conventions.md`를 따른다. 이 문서는 그 위에서 "질"을 본다.

---

## 핵심 명제

테스트의 가치는 그것이 **실행됐을 때 우리가 얻는 정보**로 측정한다.
- 실패 = 진단 (무엇이 왜 깨졌는지)
- 통과 = 보증 (이 동작은 실제로 보장된다)

정보를 주지 못하는 테스트는 통과해도 무가치하고, 잘못된 정보를 주는 테스트는 해롭다.

---

## A. 좋은 테스트의 5대 속성

모든 테스트는 아래 5개를 지향한다.

1. **진단성** — 실패 메시지와 테스트명만으로 원인이 좁혀진다. 디버거를 켜야 원인을 알면 약한 테스트다.
2. **보증성** — 통과가 "이 동작은 실제로 보장된다"를 의미한다. 구현 로직을 단언으로 복사하면 통과해도 보증이 없다.
3. **행동 검증** — 입력 → 관측 가능한 출력·부수효과를 검증한다. 내부 호출 순서나 private 상태가 아니다.
4. **결정성** — 시간·실행 순서·실제 네트워크·파일시스템 경합에 무관하게 동일한 결과를 낸다.
5. **리팩터링 내성** — 동작을 바꾸지 않는 리팩터링에서 깨지지 않는다. 깨지면 그건 구현 테스트다.

---

## B. 안티패턴 체크리스트

전수 스캔 시 각 테스트(또는 파일)에 아래 코드를 태깅한다.

| 코드 | 안티패턴 | 탐지 신호 |
|------|----------|-----------|
| AP-1 | 동어반복 (구현 복사) | 단언값이 구현의 상수/식을 그대로 복사 — 버그도 함께 복사돼 절대 안 깨짐 |
| AP-2 | 목 검증 과다 | `toHaveBeenCalled*` 류만 있고 실제 결과·상태 단언이 없음 |
| AP-3 | Flaky | 실제 `setTimeout`/실제 네트워크/실행 순서/파일시스템 경합에 의존 |
| AP-4 | 단언 없음 | smoke만 — `expect` 0개, "에러 안 나면 통과" |
| AP-5 | 과잉 명세 | 검증 의도와 무관한 필드까지 전체 객체를 고정 (거짓 양성 유발) |
| AP-6 | 거대 테스트 | 한 `it`/`test`에 다수 시나리오를 묶어 실패 시 원인 특정 불가 |
| AP-7 | 조용한 스킵 | `it.skip`/`todo`/영구 opt-in 환경가드로 커버리지 착시 |

## C. 심각도 등급

| 등급 | 의미 | 해당 안티패턴 |
|------|------|---------------|
| **P0** | 거짓 음성 — 버그인데 초록불. 안전망이 거짓말함 | AP-1, AP-4 |
| **P1** | 신호 오염 — 거짓 양성/불신으로 빨간불을 무시하게 만듦 | AP-3, AP-7 |
| **P2** | 유지보수 부채 — 동작은 검증하나 변경 비용이 큼 | AP-2, AP-5, AP-6 |

거짓 양성(멀쩡한데 빨간불)과 거짓 음성(버그인데 초록불)은 둘 다 테스트를 무가치하게 만든다. 점검의 본질은 "이 테스트가 어느 쪽 거짓을 낼 위험이 있나"를 보는 것이다.

---

## D. 판정 규칙 (이 프로젝트 합의 사항)

1. **목 검증 자체는 죄가 아니다.** Electron IPC/PTY/LSP 경계 때문에 목은 불가피하다. AP-2는 "목 호출 검증만 있고 **결과 단언이 함께 없는** 경우"에만 태깅한다. `pattern-bun-mock-conventions.md`를 따르는 목은 정상.
2. **Flaky 판정의 현실선.** `bun:test`의 가짜 타이머·`mock`은 정상으로 본다. **실제** `setTimeout`/실제 네트워크/실제 파일시스템 경합만 AP-3로 태깅한다.
3. **커버리지 공백은 범위 밖.** 이 문서와 이번 점검은 *기존 테스트의 질*만 다룬다. *없는 테스트*(미검증 핵심 모듈)는 다음 사이클로 미룬다.
4. **현재 fail/error는 안티패턴 자동분류 대상이 아니다.** 실패 케이스는 별도로 `코드 버그 / 나쁜 테스트 / 환경 문제` 3분류를 수행한다.
5. **점검은 제안만 한다.** 삭제·수정은 합의 후 별도 사이클에서 실행한다.

## 적용 체크리스트

각 테스트를 볼 때:

- [ ] 이 테스트가 깨지면 원인을 바로 알 수 있는가? (진단성)
- [ ] 통과가 실제 동작을 보증하는가, 아니면 구현을 복사했는가? (보증성 / AP-1)
- [ ] `expect`가 결과·상태를 검증하는가, 목 호출만 보는가? (AP-2, AP-4)
- [ ] 실제 시간/네트워크/FS에 의존하는가? (AP-3)
- [ ] 검증 의도와 무관한 필드까지 고정했는가? (AP-5)
- [ ] 한 케이스가 한 가지만 검증하는가? (AP-6)
- [ ] 조용히 스킵된 상태인가? (AP-7)
