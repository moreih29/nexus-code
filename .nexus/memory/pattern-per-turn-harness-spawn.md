# pattern-per-turn-harness-spawn

AI CLI 하네스를 매 턴마다 새 프로세스로 spawn하고 `--resume <sessionId>`로 이전 세션을 이어붙이는 방식으로 감독한다.

## 적용 조건 — 언제 이 패턴이 맞는가

다음 조건을 **모두** 충족할 때 이 패턴을 선택한다.

- 외부 AI CLI(예: claude-code, opencode, codex)가 세션 파일 기반 resume을 **공식 CLI 옵션**으로 지원한다.
- 하네스 프로세스의 장기 실행 안정성을 보장할 수 없다(메모리 증가, 이벤트 루프 정체, 스트림 끊김 이력).
- IDE 코어가 각 AI 응답 턴의 시작·종료를 명확히 구분해야 한다(이벤트 경계 추적, 타임아웃 적용).
- 어댑터 코드를 단순하게 유지해야 하는 1인 또는 소규모 팀 환경이다.

다음 중 하나라도 해당하면 이 패턴을 **쓰지 않는다**.

- 하네스가 resume 기능을 공식 지원하지 않아 세션 파일을 수동으로 파싱해야 한다.
- 턴당 spawn 오버헤드(수백 ms)가 해당 제품의 UX 요구사항에서 허용 불가 수준이다.
- 스트리밍 토큰을 실시간으로 렌더링해야 하는 채팅 UI를 직접 구현한다(이 경우 stream-json 장기 연결이 불가피할 수 있다).

## 장점

- **프로세스 격리**: 각 턴이 독립 프로세스로 실행되므로 메모리 누적, 좀비 스트림, 이벤트 루프 정체가 다음 턴에 파급되지 않는다.
- **공식 경로 사용**: 하네스 제공자가 지원하는 CLI 인터페이스만 사용하므로 비공개 내부 프로토콜 의존이 없다. 하네스 버전 업그레이드 시 어댑터 수정 범위가 최소화된다.
- **어댑터 구현 단순**: `spawn → stdout 수집 → exit 대기 → sessionId 저장` 구조만 구현하면 된다. 멀티플렉싱, 재연결 watchdog, 스트림 상태 머신이 불필요하다.
- **타임아웃·취소 용이**: 프로세스 단위로 `SIGTERM`을 보내면 되므로 미응답 턴을 깔끔하게 중단할 수 있다.
- **대화 이력 보장**: 대화 히스토리가 세션 파일에 기록되므로, 프로세스가 종료되어도 `--resume`으로 정확히 이어진다.

## 단점 · 트레이드오프

- **턴 간 spawn 오버헤드**: 각 턴마다 새 프로세스를 초기화하므로 수백 ms의 지연이 발생한다. 사용자가 빠른 연속 질문을 하는 시나리오에서 체감된다.
- **스트리밍 토큰 표시 제한**: 프로세스 stdout을 그대로 파이프하면 스트리밍은 가능하나, 재연결 시 이미 출력된 토큰을 재전송받지 못한다.
- **sessionId 상태 관리**: IDE가 워크스페이스별 sessionId를 영속화하고 올바른 ID를 전달하는 책임을 진다.

## 대안과 비교 — 왜 이 패턴을 선택하는가

**대안: long-lived process + stdin.write (stream-json 다중 턴)**

하네스 프로세스를 하나만 띄워두고 stdin에 새 요청을 써서 다중 턴을 처리하는 방식이다. 이 방식은 spawn 오버헤드가 없고 스트리밍 연속성이 유지된다.

그러나 이전 프로젝트(Tauri 기반)에서 이 방식을 적용한 결과, 다음 문제들이 실증되었다.

- 하네스 내부 이벤트 루프가 장시간 실행 후 정체되어 응답이 멈추는 현상 (`turn_end` 이후 무응답).
- 프로세스를 죽이지 않으면 복구가 불가능해 EventSource 재연결 watchdog을 별도로 구현해야 했다.
- `stream-json` 모드는 하네스의 비공개 내부 인터페이스에 해당해, 하네스 업데이트 시 어댑터가 무음 파손되는 사례가 반복됐다.

per-turn spawn은 spawn 오버헤드를 트레이드오프로 감수하는 대신, 위 실패 패턴을 구조적으로 차단한다.
