# pattern-harness-observer

IDE는 AI 하네스의 TUI를 대체하지 않고 hook API 또는 세션 파일 tail을 통해 읽기 전용으로 관찰해, 탭 뱃지·diff 뷰·알림 등 부가 UI만 얹는다.

## 적용 조건 — 언제 이 패턴이 맞는가

다음 조건을 **모두** 충족할 때 이 패턴을 선택한다.

- AI 하네스(claude-code, opencode, codex 등)가 자체 TUI를 가지고, 그 TUI 경험을 사용자에게 그대로 제공하는 것이 제품 약속의 일부다("네이티브 터미널 수준 AI 경험").
- IDE가 필요로 하는 것이 완전한 채팅 UI가 아니라, **하네스 상태를 IDE 맥락에서 보조하는 augmentation**이다(진행 상황 표시, 파일 변경 감지, 완료 알림).
- 하네스 프로토콜 변경에 IDE 코어가 영향받는 것을 최소화해야 한다(하네스 제공자의 내부 변경이 잦음).
- 어댑터를 플러그인 경계 밖에 격리해 코어 안정성을 유지해야 한다.

다음 중 하나라도 해당하면 이 패턴을 **쓰지 않는다**.

- 사용자가 IDE 내에서 AI와 직접 대화하는 채팅 UI를 기대한다(Cursor·Zed 스타일 자체 채팅 패널이 핵심 UX인 경우).
- 하네스가 hook API나 세션 파일 같은 관찰 인터페이스를 전혀 제공하지 않는다.
- 승인 다이얼로그를 OS 네이티브 다이얼로그로 가로채는 등 하네스 제어 흐름에 개입해야 하는 기능이 필수다.

## 장점

- **TUI 네이티브 경험 유지**: 사용자는 ITerm2·Ghostty에서 claude-code를 실행하는 것과 동일한 방식으로 AI 하네스를 사용한다. 한글 입력, 스크롤, 단축키 등 TUI 고유의 경험이 손상되지 않는다.
- **하네스 프로토콜 변경에 코어가 영향받지 않음**: 관찰 인터페이스(hooks, 세션 파일 포맷)만 어댑터 레이어에서 처리하므로, Anthropic이나 OpenAI가 내부 포맷을 변경해도 `HarnessAdapter` 인터페이스를 구현하는 어댑터 파일만 수정하면 된다. Electron 코어나 React UI는 수정 불필요.
- **플러그인 경계 원칙 준수**: 어댑터가 제품 코어 외부에 격리되어 있으므로, 하나의 하네스 어댑터 파손이 다른 하네스나 IDE 핵심 기능에 파급되지 않는다.
- **구현 범위 한정**: 자체 채팅 UI, 스트리밍 렌더러, 모델 연결 레이어를 구축하지 않아도 된다. 이전 Tauri 프로젝트에서 이 레이어를 자체 구현했다가 폐기한 교훈을 구조적으로 반영한다.

## 단점 · 트레이드오프

- **제어 개입 불가**: 하네스가 사용자 승인을 요청할 때 IDE 네이티브 다이얼로그로 가로채거나, 하네스 응답을 중간에 수정하는 기능을 구현할 수 없다. Observer는 읽을 뿐 개입하지 않는다.
- **관찰 인터페이스 의존성**: 각 하네스의 hook API 또는 세션 파일 포맷이 공식 계약으로 유지된다는 보장이 없다. 특히 codex는 세션 스키마 변동이 잦아 어댑터 지속 보수가 필요하다(Issue 5 분석).
- **실시간성 제한**: 세션 파일 tail 방식은 디스크 쓰기 주기에 따라 이벤트 감지에 수십 ms 지연이 발생할 수 있다. Hook API 방식은 실시간에 가깝지만 하네스 지원 여부에 따라 다르다.

## 관찰 메커니즘 (하네스별)

| 하네스 | 관찰 방법 | 제공 부가 UI |
|---|---|---|
| claude-code | Hooks API (PreToolUse/PostToolUse/Notification/Stop) + `.jsonl` 세션 파일 tail | 탭 뱃지, tool 호출 사이드패널, diff 뷰, OS 알림 |
| opencode | SQLite 세션 DB + 이벤트 스트림 | 위 동일 |
| codex | 세션 파일 / JSON 출력 (스키마 변동 잦음) | 위 동일, 어댑터 보수 비용 높음 |

## 대안과 비교 — 왜 이 패턴을 선택하는가

**자체 채팅 UI로 TUI 완전 대체 (Cursor·Zed 스타일)**

IDE가 AI 모델 API에 직접 연결해 자체 채팅 패널을 구현하는 방식이다. 채팅 UX의 완전한 제어권을 가지며, 하네스 TUI 한계에서 자유롭다.

그러나 이 접근은 nexus-code의 핵심 약속인 "AI 하네스가 네이티브 터미널처럼 동작한다"와 배치된다. 자체 UI를 구현하면 하네스 TUI 고유의 경험(키바인딩, 스크롤, 한글 IME)이 손실되고, 모델 API 연결·스트리밍 렌더러·토큰 관리 레이어를 자체 구축해야 한다. 이전 Tauri 프로젝트에서 이 방향을 부분적으로 시도했다가 유지 비용이 폭증했다.

**cmux 선행 사례**: `external-competitors.md`의 cmux 항목이 이 패턴의 선행 구현 사례다. cmux는 Unix 소켓 + Claude Code hooks API로 AI 이벤트를 수신해 패인에 알림·시각화 오버레이로 표시한다. TUI를 대체하지 않고 읽기 전용으로 관찰하는 방식이 cmux의 핵심이며, nexus-code는 이 접근을 IDE 컨텍스트(탭 뱃지, diff 뷰, 파일트리 연동)로 확장한다.
