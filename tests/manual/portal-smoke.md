# Portal Migration — Manual Smoke Checklist

자동 테스트(jsdom + happy-dom)가 검증할 수 없는 런타임 fidelity를 실제 Electron 빌드로 확인합니다.

## 실행 환경

```bash
bun dev
```

## 체크 항목

### (a) Terminal split — PTY 연속성

**준비**: 워크스페이스 열기 → 새 Terminal 탭 생성.

**시나리오**:
1. 좌측 터미널에서 다음 명령 실행:
   ```bash
   for i in $(seq 1 200); do echo $i; sleep 0.1; done
   ```
2. 출력이 흐르는 중 해당 탭을 우측으로 split.

**기대**:
- [ ] 좌측 터미널의 출력 스트림이 끊기지 않고 계속 흐른다.
- [ ] 좌측 xterm 화면의 그리드 크기/폰트가 정상.
- [ ] 우측 새 PTY는 독립된 새 셸이며, 별도 명령어 입력이 가능하다.
- [ ] 콘솔(DevTools)에 WebGL context loss 경고가 없다 (또는 발생해도 즉시 canvas fallback으로 복구).

### (b) Editor split — Monaco 상태 보존

**준비**: 임의 코드 파일(test4.py 등)을 좌측에 연다.

**시나리오**:
1. cursor를 특정 행(예: 30번째)에 두고 한 글자 입력하여 dirty 표시 생성.
2. scroll을 중간 위치로 이동.
3. 해당 탭을 우측으로 split.

**기대**:
- [ ] 좌측 Monaco editor의 cursor가 정확히 같은 위치에 유지.
- [ ] scroll position 유지.
- [ ] dirty 표시(탭 옆 점)가 그대로.
- [ ] 우측은 같은 파일을 보지만 자체 cursor/scroll 상태(독립).
- [ ] LSP 진단(언더라인) 표시가 양쪽 모두 정상 동작.

### (c) Nested split — 보고된 결함 재현 (핵심)

**준비**: 단일 leaf 워크스페이스.

**시나리오**:
1. 임의 파일을 열고 좌우 split → 좌·우 두 패널 생성.
2. 우측 패널의 탭을 다시 우측 상하 split → 우측 상단·하단 두 패널 생성.

**기대**:
- [ ] **빈 갭 없음**: 우측 상단과 우측 하단 사이에 회색 빈 영역이 보이지 않는다.
- [ ] 우측 상단 패널이 자기 영역을 정확히 가득 채운다.
- [ ] 우측 하단 패널이 자기 영역을 정확히 가득 채운다.
- [ ] 좌측 패널 영향 없음.
- [ ] **스크린샷 첨부 필수** (PR/이슈에).

스크린샷 비교 기준:
- 결함 시 화면(이슈 등록 시 첨부): 우측 상단과 하단 사이에 ~80–120px 빈 갭.
- 통과 화면: 갭 0px.

### (d) 워크스페이스 스위치 — 양쪽 상태 보존

**준비**: 두 개의 워크스페이스(예: A, B)를 동시에 연다.

**시나리오**:
1. 워크스페이스 A에서 Terminal split + 출력 흘림.
2. 워크스페이스 B에서 Editor split + 코드 편집.
3. A → B → A → B 왕복 4회.

**기대**:
- [ ] 매 스위치마다 활성 워크스페이스의 split 레이아웃 즉시 표시.
- [ ] A의 PTY 출력이 비활성 동안에도 끊기지 않고 계속 흐른다 (백그라운드에서 누적).
- [ ] A로 돌아오면 누적 출력이 그대로 보인다(또는 따라잡힌다).
- [ ] B의 dirty 표시·cursor·scroll이 스위치 후에도 보존.
- [ ] 워크스페이스 슬롯 크기가 즉시 reflow되어 xterm/Monaco가 적절히 fit (잘림이나 빈 영역 없음).

## 결과 기록

각 항목 옆 `[ ]`를 `[x]`(PASS) 또는 `[FAIL: 원인]`으로 채워 PR 또는 이슈에 첨부합니다. `(c) nested split` 스크린샷은 필수 첨부.

실패 항목 발생 시:
- 회귀로 간주, 별도 이슈로 분리하거나 본 사이클에서 추가 수정 → 재실행.

## 알려진 제약

- WebGL context loss는 GPU 드라이버 환경에 따라 재현 빈도가 다름. fallback 미작동 시 별도 follow-up 이슈로 등록.
- xterm FitAddon 재계산은 ResizeObserver 의존 — 워크스페이스 스위치 시점에 reflow가 한 프레임 늦을 수 있음(허용).
